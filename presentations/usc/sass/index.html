What is SASS?

SASS is a meta-language on top of CSS thatâ€™s used to describe the style of a document cleanly and structurally, with more power than flat CSS allows. SASS both provides a simpler, more elegant syntax for CSS and implements various features that are useful for creating manageable stylesheets.

English version

Write stylesheets that are readable by humans but compile for computers to read quickly.

You can use variables and mixins for common written items, separate files for themes, modules, styleguides.

SASS vs SCSS

SCSS extends SASS to still use braces and semi-colons like CSS


// scss
p {
  margin: 0;
  padding: 0;
}

// sass
p
  margin: 0
  padding: 0

For this discussion we will focus on SCSS

SASS syntax comes from Ruby and Python whereas SCSS is a superset of CSS so any valid CSS document is valid SCSS.

So why use this?!

  • CSS preprocessors are not new (anymore)
    • There are several books on SASS
  • The web community is already using them
    • Frameworks for SASS
    • Countless blogs about SASS
  • Mixins & Variables
  • SCSS is CSS so you can start using it right away and optimize later

Does this benefit front-end development?

Yes! That's why I made this presentation.

SASS allows front-end developers to

  • Separate CSS based on type or usage
  • Build reusable components/modules
  • Deploy sites faster
  • Create sprites on the fly from a folder*

    * using Compass

Separate: we can make files for styleguides, variables, commonly used elements (mixins)

Modules: reuse snippets of code and maintain them from one source rather than copy/paste and hope to find for changes/updates

Deploy: using frameworks we can write less code quickly but maintain global and local requirements more effectively

Sprites: anyone create a sprite? And then need to change it?

Developer Benefits

Reduces server calls

  • Old Method (4 server calls)
  • New Method (1 server call) 
    
// file.scss compiles to file.css
@import 'styleguide';
@import 'modules';
@import 'component';
@import 'plugin';
@import in this instance is run prior to deployment so it generates a file vs @import on the server is actually another server call

Compile on the Fly

  • Smaller file size for faster loading
  • Maintain working SCSS files that are human readable
  • Divide your project into modular development and reusable components
  • Link countless framework files, SASS only includes the elements used

Smart Compilation

SASS will not only compile on the fly but it will group elements in a smart way, not just stack what you write


//less
.module {
  margin: 0;
  padding: 1em;
  background: white;
  color: black;
}
.module-alert {
  .module();
  background: red;
  color: white;
}

						

//sass
.module {
  margin: 0;
  padding: 1em;
  background: white;
  color: black;
}
.module-alert {
  @extend .module;
  background: red;
  color: white;
}
Here you can see the markup is basically the same with each codebase's nuance in code of css variable for LESS vs @extend for sass.

Compilation Output


//less
.module {
  margin: 0;
  padding: 1em;
  background: white;
  color: black;
}
.module-alert {
  margin: 0;
  padding: 1em;
  background: white;
  color: black;
  background: red;
  color: white;
}

						

//sass
.module, .module-alert {
  margin: 0;
  padding: 1em;
  background: white;
  color: black;
}

.module-alert {
  background: red;
  color: white;
}

Rather than duplicate the classes, SASS will combine elements that are common and separate the nuances

And you can put the '@extend' anywhere in the declaration!

The Less version puts the shared declaration in both styles - it repeats them. Whereas SASS scans the document and combines the shared elements in one declaration and then writes the individual declarations afterwards.

This is important since the order you write them is not important - well it is for best practices - but SASS will fix it for you if CSS is not in your main wheelhouse.

Output Styles

There are several output styles that you can set in your config.rb

  • Nested
  • Expanded
  • Compact
  • Compressed

Why does Output matter?

I took some files written in SCSS:

  • Sample file 1
    • Nested: 34kb
    • Compressed: 23kb
    • Savings: 32%
  • Sample file 2
    • Nested: 63kb
    • Compressed: 39kb
    • Savings: 38%

Where is that style coming from ? (!!!!!)

You don't have to search, just add line comments


// config.rb
line_comments = true
output_style = :expanded
						

/* line 1, ../sass/screen.scss */
ol {
  margin: 0;
}
/* line 3, ../sass/screen.scss */
ol li {
  list-style: none;
}
/* line 5, ../sass/screen.scss */
ol li p {
  color: #990000;
}

Design Benefits

Variables

One of the best reasons to use SASS.


// variables
$color-cardinal: #990000;
						

// scss
header {
  background: $color-cardinal;
}

// css
header {
  background: #990000;
}

Variables: set colors or options as, well, a variable. Write the variable once and re-use it throughout your code. Have a change, update one place and save/recompile.

Mixins

Mixins allow us to write a function once, and reuse it as needed


// mixin
@mixin rounded($radius:5px) {
  -moz-border-radius: $radius;
  -webkit-border-radius: $radius;
  -ms-border-radius: $radius;
  border-radius: $radius;
}

// scss
.module {
  @include rounded();
}
.module-alt {
  @inlcude rounded(10px);
}
						

// css
.module {
  -moz-border-radius: 5px;
  -webkit-border-radius: 5px;
  -ms-border-radius: 5px;
  border-radius: 5px;
}
.module-alt {
  -moz-border-radius: 10px;
  -webkit-border-radius: 10px;
  -ms-border-radius: 10px;
  border-radius: 10px;
}

Frameworks

There are many supported frameworks available with documentation for usage:

  • Compass: adds nested rules, variables, mixins, selector inheritance, and more.
  • Susy: Responsive grids for Compass
  • Bourbon: simple and lightweight mixin library for SASS

Styleguides

Modular development means we can separate template, module and styleguide files for ease of editing.


@import '../global/sass/compass';
@import '../global/sass/mixins';
@import 'variables';
@import 'modules';
@import 'grid';
@import 'styleguide';
						

// modules
@import '../global/sass/modules';
/* projet module items */

Include global elements in your local elements or don't write them specifically for that project. No more having to overwrite global styles.
